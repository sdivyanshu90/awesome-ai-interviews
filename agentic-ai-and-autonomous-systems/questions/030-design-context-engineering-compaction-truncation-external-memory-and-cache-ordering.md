### Q: Design context engineering for a long-horizon agent: compaction checkpoints, tool-result truncation, external memory, and cache-friendly ordering.
* **Difficulty:** Senior
* **Category:** Architecture
* **The 10-Second Pitch:** Treat the context window as a managed cache over durable state, not a transcript. Each step the harness assembles a stable prefix (system prompt, tool schemas), an append-only history whose aged tool results are truncated to stubs, and the current working state; at a token threshold it compacts history into a summary that must preserve constraints, decisions, and todo state, spilling everything else to the file system for re-read. Append-only ordering keeps prompt-cache hits high, which dominates cost and latency across hundreds of steps.
* **The Deep Dive:** The core design decision is what the model sees at step N, and it is a harness decision, not a model behavior. A workable layout is a fixed prefix (system prompt, tool schemas, environment facts) that never changes mid-run; a durable working-state block (task goal, hard constraints, decision log, todo list with per-item status); the compacted or truncated action history; and the most recent observations verbatim. Old tool results are the main token sink with the lowest marginal value: a file read from forty steps ago informed a decision already recorded, so after a freshness horizon the harness replaces its body with a stub — "read src/auth.py, 212 lines, token validation lives here" — plus the path needed to re-fetch it. The action record is kept so the model knows what it already tried; only the payload expires.

  Compaction is the riskier operation because it is lossy. When assembled context crosses a checkpoint threshold (commonly 70 to 80 percent of the window, leaving headroom for the next burst of observations), the harness rewrites history into a summary. The failure mode is uniform summarization: constraints, decisions with rationale, open todo items, files touched, and unresolved errors must survive verbatim, while exploration chatter and stale payloads may be dropped. A compaction that collapses "the change must stay backward compatible; approach B was rejected because of the v1 clients" into "worked on the migration" produces goal drift ten steps later that no amount of model quality repairs.

  ```text
  [static prefix: system + tool schemas]        never mutated; cache-hot
  [working state: goal, constraints, todo]      rewritten only at checkpoints
  [compacted history: decisions + stubs]        lossy; constraints preserved verbatim
  [recent window: last K steps, full results]   verbatim observations
  ```

  The file system serves as unbounded external memory: the agent writes notes, findings, and its todo list to files and restores them via paths kept in the stub layer, so truncation is recoverable rather than destructive; re-appending the todo list near the context tail each step also counters positional degradation. Ordering is dictated by prompt caching: caches hit only on an identical prefix, so the prefix must be append-only and deterministic — no timestamps in the system prompt, stable tool-schema ordering, no in-place edits to earlier turns. Failure trace: a harness stamps the current time into the system prompt every step; every request misses cache from token zero, and since cached input tokens are roughly an order of magnitude cheaper than uncached on major providers, cost and time-to-first-token blow up several-fold. The falsifiable test for context rot: insert a fixed probe (restate the three hard constraints) at scripted step depths across an eval suite and plot accuracy against step count and compaction count — a healthy harness shows flat probe accuracy; a degrading one shows a monotone decline that pinpoints whether truncation or compaction is eating state.
* **Production Reality & Tradeoffs:** Compaction bugs are silent — the agent continues confidently with amputated constraints, so keep the full raw trajectory in a separate log for replay and for diffing against the compacted view. Truncation must age out the oldest results, never the newest; an off-by-one here deletes the observation the model is currently reasoning about. Instrument cache hit rate, tokens per step, and compaction frequency as first-class metrics: a cache-hit collapse is usually an accidental prefix mutation. Aggressive truncation trades occasional re-read tool calls for head-room — usually a good trade, since re-reads are targeted while retained payloads tax every subsequent step.
* **Red Flag:** Treating a bigger context window as a substitute for the harness — appending everything until overflow, then summarizing once in a panic. Quality degrades well before the hard limit while per-step cost and latency grow with every retained token, and a single unstructured emergency summary is precisely the lossy compaction that destroys constraint and todo state.
