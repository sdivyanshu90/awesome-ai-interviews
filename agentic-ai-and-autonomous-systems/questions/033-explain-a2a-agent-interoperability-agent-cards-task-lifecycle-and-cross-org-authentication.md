### Q: Explain A2A-style agent interoperability: agent cards, task lifecycle, and cross-organization authentication.
* **Difficulty:** Principal
* **Category:** Multi-Agent Systems
* **The 10-Second Pitch:** A2A (Agent2Agent, a 2025 Google-initiated protocol now under the Linux Foundation) standardizes how independently built agents delegate work: a client discovers a remote agent through its published agent card, opens a task that moves through a defined lifecycle with streamed updates, and authenticates with ordinary web credentials declared in the card. It complements MCP — MCP is the agent-to-tool contract, A2A the agent-to-agent contract — and its defining property is opacity: you observe task states and artifacts, never the remote agent's internal reasoning, tools, or memory.
* **The Deep Dive:** Discovery starts with the agent card, a JSON document served from a well-known HTTPS path on the remote agent's domain. The card advertises identity, the service endpoint, a list of skills, supported input and output modalities, whether streaming and push notifications are available, and the security schemes the endpoint requires. A client fetches cards, matches a subtask against advertised skills, and picks a delegate. Critically, a card is an advertisement, not a proof: nothing in the protocol verifies that the agent behind the endpoint can do what the card claims. Trust must come from outside — curated registries, allowlists, contracts — and a spoofed or inflated card is the A2A analogue of a malicious tool description.

  Delegation is structured as a task, not a chat. The client sends a message composed of parts (text, files, structured data) over JSON-RPC on HTTP; the server mints a task id and the task advances through explicit states: submitted, working, input-required (the remote agent needs clarification, enabling multi-turn exchanges), and terminal states completed, failed, or canceled. Outputs arrive as artifacts, also built from parts. For long-running work the client either holds a server-sent-events stream carrying incremental status and artifact updates, or registers a webhook so a task that outlives any single connection can still report back. This shape is deliberate information hiding at an organizational boundary: the remote agent may internally run a fifty-step loop over its own tools, but the caller sees only state transitions and results — exactly what makes cross-vendor composition possible, and what makes debugging a partner's agent impossible from your side.

  Authentication is delegated to standard web mechanisms — OAuth 2.0, OpenID Connect, API keys, or mutual TLS, declared per-endpoint in the card and enforced at the HTTP layer, never inside message payloads. The hard problem is authorization across the boundary. Failure trace: org A's support agent delegates "refund order 4417" to org B's billing agent using A's long-lived, broadly scoped service credential. B can verify that A called, but not which end user authorized what; when a prompt-injected document in A's context fabricates refund tasks, B executes them under a perfectly valid credential. The confused deputy is structural; the fix is user-scoped delegation — token-exchange or on-behalf-of flows attenuating authority to this user and this task — plus B enforcing its own policy gates regardless of how trusted the caller looks.

  When does a protocol beat shared state? Inside one org and one runtime, a single orchestrator with shared state is strictly richer: full-bandwidth context, one trace, no serialization loss. A protocol wins when agents cross ownership, trust, or lifecycle boundaries — different vendors, codebases, deploy cadences — because a narrow contract decouples releases and is auditable. A falsifiable interop test: swap the remote agent for a different implementation serving an equivalent card; if the client needs code changes beyond configuration, that was a point integration, not interoperability. Maturity caveats are real as of early 2026: the spec is still evolving, production cross-organization deployments are scarce, discovery and registries are immature, and delegated-authorization semantics are not standardized.
* **Production Reality & Tradeoffs:** Every delegation adds a remote hop with someone else's queue behind it, so budget for latency, retries, and idempotent task submission; a resubmitted refund task must not execute twice. Version skew between a cached card and actual behavior causes quiet failures, and observability stops at the task boundary — contract tests, SLAs, and correlation ids replace step-through debugging. Meter spend per partner.
* **Red Flag:** Framing A2A and MCP as rivals, or wiring a remote agent in as a synchronous tool — tasks are long-running, stateful, and may demand mid-flight input. Equally disqualifying: trusting the agent card as verified capability, or passing a broad service credential across the boundary instead of user-scoped delegated authority.
