### Q: Explain native tool calling under the hood: schema injection, constrained emission, tool choice, parallel calls, and failure modes.
* **Difficulty:** Senior
* **Category:** Systems
* **The 10-Second Pitch:** Tool definitions are serialized into a system-prompt region in the exact format the model was post-trained on; the model emits a structured call block that the API parses into typed call objects; strict modes constrain decoding to the schema grammar; `tool_choice` forces or forbids calling; parallel calls are matched to results by id, not order. Failures are semantic — hallucinated names, placeholder arguments, over-calling — which schemas alone cannot prevent.
* **The Deep Dive:** The model has no runtime linkage to your functions. When a request includes tool definitions (name, description, JSON Schema parameters), the provider renders them into a reserved region of the context in a training-matched serialization, which is why definitions bill as input tokens and why description wording measurably changes call behavior — the description is prompt, not metadata. During post-training the model learned to enter a tool-call mode delimited by special tokens and to emit a name plus JSON arguments there; the API parses that emission into a typed block (a `tool_use` block or `tool_calls` entry) and sets a stop reason telling the harness the turn ended on a call, not an answer.

  Emission can be constrained. In strict or structured modes the provider compiles the parameter schema into a grammar and masks invalid tokens at each decoding step, guaranteeing syntactically valid JSON that conforms to a supported schema subset — at the cost of first-use grammar-compilation latency and restrictions on schema features. Constrained decoding guarantees shape, never truth: a strictly valid call can still target the wrong record. `tool_choice` steers the same machinery: auto lets the model decide, none forbids calls, required (or any) forces some call, and naming a specific tool forces that one — useful for routers and extraction, dangerous as a default because a forced model will fabricate arguments for questions the tool cannot answer.

  The harness contract for parallel calls is strict bookkeeping. One assistant turn may contain several call blocks, each with a unique id; the harness may execute them concurrently, then must return exactly one result per id in the next message. Matching is by id rather than position, but every id must be answered — including failures, returned as error-flagged results — before the model continues.

  ```text
  assistant turn: tool_use id=A search{"q":"invoice 4412"}
                  tool_use id=B get_account{"account_id":"ac_9"}
  next message:   tool_result id=B {...}   tool_result id=A {...}
                  order free, coverage mandatory: every id answered exactly once
  ```

  Failure modes divide into what constraints catch and what they cannot. They catch malformed JSON and schema violations. They do not catch: hallucinated tool names (calling a tool not offered — reject and reflect a corrective error back rather than crashing), placeholder or fabricated arguments (an id the model invented), over-calling (tool spam where a direct answer sufficed, often induced by a forced `tool_choice`), under-calling (answering from parametric memory instead of querying), and error loops (retrying an identical failing call). Falsifiable test: force a named tool and ask an unrelated question — you will get a schema-valid, semantically meaningless call, demonstrating that emission is constrained while grounding is not. Contrast with prompt-parsed ReAct-style actions: regex-extracted Thought/Action text has no grammar guarantee, drifts in format under long contexts, and is spoofable by tool output containing an Observation-shaped line — native calling moves parsing from your regex into the provider's constrained decoder.
* **Production Reality & Tradeoffs:** Definitions ride on every call, so put them in the cached prefix and prune aggressively — selection accuracy degrades as the tool count grows toward dozens. Disable parallel calls for non-idempotent tools, because concurrent side effects plus one failed branch leaves partial state the model must be told about explicitly. Providers differ on strict-mode schema subsets, result formats, and whether unanswered ids hard-error, so harnesses need a provider abstraction plus per-tool metrics: invalid-argument rate, hallucinated-name rate, and calls-per-task as regression signals across model upgrades.
* **Red Flag:** Believing strict schema mode makes arguments correct — it guarantees JSON shape while the model can still pass a fabricated customer id — or executing whatever name the model emits without checking it against the offered tool list.
