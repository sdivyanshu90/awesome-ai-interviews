### Q: When should you spawn subagents for context isolation versus one long context?
* **Difficulty:** Senior
* **Category:** Multi-Agent Systems
* **The 10-Second Pitch:** Spawn a subagent when a subtask is context-polluting but its product compresses well — broad searches, bulk file reads, log triage — so raw exploration happens in a disposable context and only a distilled result returns to the orchestrator. Keep one context when steps are tightly coupled or share mutable state, because the hand-off summary becomes a lossy bottleneck. Each spawn re-pays the system prompt and tool schemas, so isolation must save more junk tokens than the overhead costs.
* **The Deep Dive:** The rationale is information hiding, not role-play. An orchestrator that answers "which of these forty modules implements retry logic" by reading files inline drags every raw dump into its own history, where it taxes every subsequent decision and accelerates compaction — even though only one conclusion mattered. A subagent is a function call at the context level: it starts from a clean context containing only its task spec, burns whatever tokens the exploration needs in its own window, and returns a bounded result; its locals are then freed. The orchestrator's context stays composed of decisions and distilled facts, exactly the material long-horizon control depends on. Fan-out reads are therefore the canonical case: research, code search, doc triage, and evaluation sweeps compress hundreds of raw tokens into each surviving conclusion token.

  The hand-off is a contract in both directions, and most subagent failures are contract failures. Downward, the spec must be self-contained — goal, constraints, definitions, expected output schema, and stop conditions — because the subagent sees none of the parent conversation; an underspecified spec yields a confident answer to the wrong question. Upward, the result must be a bounded, structured report: conclusions, evidence pointers (file paths, ids) rather than payloads, confidence and gaps. Parallel fan-out follows naturally: independent subtasks run concurrently and an aggregation step reconciles the reports — noting that same-model, same-prompt workers fail in correlated ways, so disagreement between two workers is signal but agreement is weaker evidence than it looks. Token economics gate all of this: every spawn duplicates the system prompt and tool schemas (often thousands of tokens), forfeits the parent's warm prompt-cache lineage, and adds coordination turns. Isolation is profitable when expected exploration tokens far exceed spec plus report plus overhead — a good heuristic is that read-heavy, write-light subtasks isolate well, and a subtask whose full nuance fits in a paragraph should not be delegated at all.

  Subagents make things worse in two structural cases. Tight sequential coupling: when step k+1 needs the full texture of step k — a design evolving under negotiation, a debugging session where hypotheses shift line by line — forcing each step through a summary plays telephone, and the orchestrator accumulates degraded state instead of clean state. Shared mutable state: two parallel subagents editing overlapping files each hold a stale view of the other's changes. Failure trace: a refactor is split across three parallel subagents by directory; each renames the shared interface slightly differently, each reports success because its own slice compiles, and integration fails with conflicts no agent can see from inside its sandbox — the fix is to serialize writes through one context (or one owner per file), keeping parallelism for reads. The falsifiable test: run a fixed task suite in orchestrator-only and subagent configurations and compare success rate, end-to-end tokens, and orchestrator context growth; if the subagent configuration does not win on read-heavy tasks or loses on tightly coupled ones, the boundary is drawn wrong.
* **Production Reality & Tradeoffs:** Report quality is the weak link — a subagent that returns vague prose poisons the orchestrator more subtly than raw dumps would, so enforce report schemas and evaluate them like any tool output. Debugging spans agent boundaries: propagate trace ids so a bad final answer can be attributed to a bad sub-report. Cap recursion depth and concurrent children, meter child spend against the parent run budget, and expect latency: fan-out helps wall-clock only when subtasks are genuinely parallel, while sequential delegation adds round trips. Worktree- or sandbox-per-child isolation contains write conflicts but adds merge cost.
* **Red Flag:** Splitting into "planner, critic, researcher" personas that all share one context anyway — or delegating for org-chart aesthetics rather than information flow. If a candidate cannot say what tokens the boundary keeps out of the orchestrator and what the child's report schema is, the subagents are overhead cosplaying as architecture.
