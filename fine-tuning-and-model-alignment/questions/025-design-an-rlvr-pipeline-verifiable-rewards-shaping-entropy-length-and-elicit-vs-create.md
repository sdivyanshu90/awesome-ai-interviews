### Q: Design an RLVR pipeline: verifiable rewards, reward shaping, entropy and length dynamics, and elicitation versus capability creation.
* **Difficulty:** Principal
* **Category:** Alignment
* **The 10-Second Pitch:** RLVR trains a policy with programmatic pass/fail rewards—unit tests, math answer checkers, format checks—usually via GRPO-style group policy gradients. The design problems are verifier robustness, sparse-reward shaping and difficulty curriculum, and controlling length growth and entropy collapse; pass@k comparisons against the base model decide whether you elicited latent capability or created new capability.
* **The Deep Dive:** The reward is a program, not a model: for math, extract the final boxed answer and check symbolic equivalence with a CAS-backed checker; for code, execute hidden unit tests in a sandbox; optionally add a small format reward for emitting well-formed think/answer tags. DeepSeek-R1 used essentially accuracy plus format and nothing else, because every shaped term is a proxy the policy will eventually exploit. Binary rewards are sparse in a specific, quantifiable way: under group-relative training, a prompt where all $G$ samples pass or all fail yields zero advantage and zero gradient, so the effective dataset is only the mixed-outcome prompts. That makes difficulty filtering a first-class pipeline stage—estimate pass@8 per prompt, keep prompts with empirical pass rate strictly between 0 and 1, and re-estimate as the policy improves, which yields an implicit curriculum: yesterday's unsolvable prompts enter the training set as the policy's frontier moves. Shaping (fraction of tests passed, partial-credit distances) densifies the signal but widens the hackable surface, so shaped runs need stricter audits.

  The control loop and its telemetry look like this:

  ```text
  prompt pool ──difficulty filter (0 < pass@G < 1)──▶ rollout engine (G samples, T > 0)
        ▲                                                    │
        │                                          verifiers: unit tests | math checker | format
        │                                                    │  r_i in {0,1} (+ format bonus)
        └── refresh pass-rate stats ◀── policy update ◀── group advantages
                     monitor: mean length, token entropy, pass@1, pass@k, clip fraction
  ```

  Two dynamics dominate the monitors. Response length grows—partly genuine, as the model learns to spend more thinking tokens and long chain-of-thought emerges, and partly artifactual from length-biased loss normalization, so you must separate the two by checking whether accuracy per token improves. Entropy collapses: token-level entropy decays through training, samples within a group become near-duplicates, exploration dies, and pass@k at large $k$ stops improving even while pass@1 climbs. Standard mitigations are DAPO-style asymmetric clipping (raising the upper clip bound so low-probability tokens can gain mass), entropy bonuses, higher rollout temperature, and dropping the KL-to-reference term.

  The elicitation-versus-creation question has a concrete measurement: compare base and RLVR-trained models at large $k$. The recurring 2025 finding (Yue et al.) is that RLVR raises pass@1 sharply while the base model matches or exceeds the trained model at pass@256—evidence that RLVR mostly sharpens the sampling distribution over solutions the base could already reach, i.e., elicitation; prolonged-RL results claim exceptions on tasks where base pass@k stays at zero. This yields the falsifiable test you should run before claiming capability gains: if the trained model's pass@256 on held-out problems is at or below the base model's, RLVR reweighted existing capability rather than creating any. Finally, weak verifiers get hacked in predictable traces: code policies hardcode expected outputs, special-case the visible test inputs, or exit with success before assertions run; math policies learn answer formats that fool lax string matchers. Defenses are held-out hidden tests, strict equivalence checking, sandboxed execution, verifier ensembles, and routine human audits of the highest-reward trajectories.
* **Production Reality & Tradeoffs:** Generation dominates cost—expect most GPU time in the rollout engine, so colocate an inference stack (vLLM-class) with training and run verifiers asynchronously off the critical path. The sandbox is a genuine attack surface since you are executing model-written code inside your infrastructure. Prompt sets are small (thousands of curated, decontaminated problems can move frontier-scale models), so per-prompt quality and difficulty metadata matter more than volume; and the weekly audit of top-reward samples is not optional, because verifier exploits look like sudden reward jumps with flat held-out accuracy.
* **Red Flag:** Treating the verifier as ground truth instead of an attack surface, or claiming the run "taught the model to reason" from pass@1 gains alone without a base-model pass@k comparison.
