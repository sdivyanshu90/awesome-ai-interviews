### Q: Compare contextual retrieval (index-time LLM chunk context), document-summary prepending, and late chunking.
* **Difficulty:** Senior
* **Category:** RAG
* **The 10-Second Pitch:** All three attack the same defect—chunks embedded without document context are ambiguous—either by paying an LLM at index time to write per-chunk situating text, prepending one cheap document summary to every chunk, or embedding the whole document first and pooling per-chunk token vectors afterward.
* **The Deep Dive:** Contextual retrieval, popularized by Anthropic in late 2024, runs an LLM over each chunk with the full document in the prompt and asks for a short situating preamble ("This chunk is from ACME's Q2 2023 filing and discusses revenue growth..."), then indexes preamble plus chunk in both the embedding and the BM25 index. Anthropic reported the top-20 retrieval failure rate dropping by roughly a third with contextual embeddings alone, about half when combined with contextual BM25, and about two thirds with a reranker added. The cost is one LLM pass per chunk with the document as context; with prompt caching the document tokens are re-read cheaply across its chunks, bringing the reported cost to around one dollar per million document tokens—a one-time index cost amortized over every future query, which is exactly the right side of the ledger for read-heavy corpora.

  Document-summary prepending is the budget version: generate one summary per document and prepend the same summary (or just title, section path, and date) to every chunk before embedding. It costs one LLM call per document instead of per chunk and fixes coarse ambiguity—which company, which year—but cannot resolve chunk-specific references, and identical prefixes push all of a document's chunk embeddings toward each other, which can hurt within-document discrimination. Late chunking inverts the pipeline: a long-context embedding model encodes the entire document once, so every token's representation is conditioned on the full document through attention, and chunk vectors are then produced by pooling token embeddings within each chunk's boundary. It needs zero LLM calls and no synthetic text, but it requires an embedder whose context window covers the document (or a windowed approximation for longer ones), and it produces single-vector chunks whose gains are typically smaller than per-chunk LLM context on anaphora-heavy corpora.

  Citation stability separates them sharply. Late chunking never alters stored text, so span offsets and citations remain exact. Contextual retrieval and summary prepending inject synthetic text; if you cite from the indexed string, users see sentences the author never wrote. The invariant: store retrieval text and citation text separately, and cite only from the original span. Freshness costs also differ: editing a document invalidates every chunk's LLM context and the summary, and late chunking must re-embed the whole document, so all three pay document-scoped reindexing—contextual retrieval just pays the largest bill per edit. The falsifiable test: partition your eval chunks into those whose raw text already names the key entity and those that reference it only anaphorically ("the company", "this model"). Contextual retrieval's gains should concentrate heavily in the anaphoric partition; if improvement is uniform across both, the lift is likely coming from longer indexed text rather than disambiguation, and cheap summary prepending would capture most of it.
* **Production Reality & Tradeoffs:** Decide on churn and budget: for a slowly-changing, read-heavy knowledge base with citation requirements, contextual retrieval plus hybrid BM25 is usually worth the index-time spend; for high-churn corpora the per-edit re-contextualization cost dominates and late chunking or summary prepending wins; for corpora that fit a long-context embedder, late chunking is nearly free to try. Version the context-generation prompt as part of the index version, because changing it silently shifts the embedding distribution and breaks A/B comparability.
* **Red Flag:** Citing text from the augmented chunk so users see LLM-invented context presented as the source document, or re-contextualizing only new chunks after an edit so old chunks carry stale situating text that contradicts the current document.
