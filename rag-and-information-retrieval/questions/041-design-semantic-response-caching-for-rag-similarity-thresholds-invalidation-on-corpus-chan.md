### Q: Design semantic response caching for RAG: similarity thresholds, invalidation on corpus change, and per-tenant isolation.
* **Difficulty:** Senior
* **Category:** System Design
* **The 10-Second Pitch:** Cache full RAG answers keyed by query embedding and serve a hit when a new query is close enough and every compatibility key matches—tenant, ACL fingerprint, index version, model and prompt versions—with invalidation wired to document updates, because a semantically wrong or stale or cross-tenant hit costs far more than the generation it saved.
* **The Deep Dive:** The hit path embeds the normalized query, runs ANN over cached entries, and accepts the nearest entry only if cosine similarity clears a threshold $\tau$ and the entry's compatibility keys match exactly. The threshold is the dangerous part: embedding similarity is a paraphrase detector, not an equivalence test. Minimal pairs—"Q3 2025 revenue" versus "Q4 2025 revenue", "enable SSO" versus "disable SSO", same question about a different customer—routinely score above $0.9$, and negation is notoriously weakly encoded. A workable design layers guards: a conservative $\tau$ around $0.95$ tuned per intent class, an exact-match requirement on extracted entities, numbers, and dates between the new query and the cached one, and optionally a cheap cross-encoder verification for borderline hits. Every guard trades hit rate for correctness; the tuning target is a measured false-hit rate, not a similarity number that feels high.

  Staleness is governed by construction, not TTL alone. Each cache entry records the retrieval index version and the exact set of source document IDs and versions its answer cited. Two invalidation triggers follow: a bump of the global index or embedding or prompt-template version invalidates wholesale (the version is part of the key, so old entries simply stop matching), and a document update or deletion invalidates precisely, via an inverted map from document ID to the cache entries that cited it, processed from the ingestion pipeline's change feed. A TTL of hours to a day remains as backstop for missed events. Without the document-level map, a corrected price or retracted policy keeps being served from cache for the full TTL—an incident, not a performance detail.

  Isolation is absolute: the cache namespace is keyed by tenant, and within a tenant by an ACL fingerprint—a hash of the permission groups that gated the retrieval that produced the answer. A user whose groups differ from the fingerprint misses, even on an identical query, because the cached answer may embed evidence from documents that user cannot read; serving it is a data leak that bypasses every retrieval-time ACL check you built. This shreds hit rates in fine-grained-permission deployments, and the honest response is to accept it or cache only over provably public corpora—never to widen the key. The falsifiable test ships with the cache: an adversarial suite of minimal-pair queries (entity swaps, date swaps, negations) asserting misses, a replayed document-update event asserting dependent entries invalidate within the SLA, and a cross-tenant probe asserting a tenant B user can never observe a tenant A entry. If the entity-swap set produces more than about $1$ percent false hits at your chosen $\tau$, the threshold-only design is falsified and the entity guard is mandatory.
* **The economics that justify all this machinery:** is deliberately simple arithmetic. At one million queries per day, a 25 percent semantic hit rate saving roughly $2\,000$ generation tokens per hit avoids about 500M tokens daily, while the lookup adds only an embedding call and an ANN probe, typically 10 to 20 ms. The savings are real but asymmetric: one publicized wrong or leaked answer can erase the trust the whole system depends on.
* **Production Reality & Tradeoffs:** Expect semantic hit rates of 20 to 40 percent on support-style traffic and near zero on long-tail analytical queries, so measure before building; monitor hit rate, false-hit reports, invalidation lag, and per-tenant distributions as separate SLOs; and shadow-serve the cache first, comparing cached versus fresh answers offline to estimate the false-hit rate on live traffic before any user sees a cached response. Keep entries immutable and log which cache entry served each response for auditability.
* **Red Flag:** A global cache namespace shared across tenants or ACL scopes "because the queries are the same," or a cache with no linkage to index versions and document updates—both turn a cost optimization into a correctness and security incident generator.
