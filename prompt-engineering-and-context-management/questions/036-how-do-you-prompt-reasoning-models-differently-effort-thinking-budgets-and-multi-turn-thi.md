### Q: How do you prompt reasoning models differently: effort and thinking budgets, when explicit chain-of-thought scaffolds hurt, and multi-turn thinking blocks.
* **Difficulty:** Senior
* **Category:** Reasoning
* **The 10-Second Pitch:** Reasoning models expose test-time compute as an API knob — reasoning-effort levels or an explicit thinking-token budget — so you steer depth there, keep the prompt a plain task statement with a clear output contract, and drop hand-written chain-of-thought scaffolds that fight the trained deliberation policy. In tool loops, return thinking blocks unmodified within the current turn.
* **The Deep Dive:** The knobs replace prompt tricks. OpenAI-style reasoning models take an effort setting (minimal through high); Anthropic extended thinking takes an explicit budget of thinking tokens; Gemini exposes a similar thinking budget. All buy serial deliberation before the visible answer: accuracy rises with budget on hard math, code, and planning tasks with strongly diminishing returns, while cost and latency grow roughly linearly because thinking tokens bill as output tokens. So calibrate per task class — route simple classification at minimal effort and hard synthesis at high effort — rather than fixing one global value and tuning wording around it.

  Why scaffolds hurt: these models were post-trained with reinforcement learning on verifiable rewards (RLVR), so they already learned to explore, backtrack, and self-verify inside a thinking channel. An appended "think step by step" is at best redundant. Worse are hand-rolled tree-of-thought harnesses that split the problem into many short sampled calls with voting: they fragment deliberation across call boundaries the model cannot backtrack over, and they typically lose to a single high-effort call at matched total compute. Few-shot chain-of-thought exemplars can additionally anchor the model to the exemplars' shallow reasoning style; DeepSeek-R1's own documentation recommends zero-shot direct prompting because few-shot prompting degraded its benchmark scores. What still belongs in the prompt: the task, constraints, definitions, tool inventory, and output format — instruction following is intact; it is the reasoning procedure you should stop scripting.

  Multi-turn thinking blocks: in a tool-use loop the assistant's turn is not finished until tool results return, so you must send the assistant message back exactly as received — thinking blocks and their signatures included. Stripping or editing them fails signature validation on Anthropic's API or severs the plan the model laid down before issuing the tool call. Across completed turns, providers strip prior-turn thinking from context automatically (it is neither re-billed nor part of the served prefix), which keeps long histories cache-friendly; but toggling thinking on or off, or changing its budget, changes the prefix and invalidates prompt caches, so fix those settings per session. Falsifiable test: on a frozen eval, run (a) a plain prompt at medium effort versus the same prompt plus a CoT scaffold or ToT harness at matched compute, and (b) a tool loop that preserves versus strips thinking blocks. If the scaffolded or stripped variants match or beat the clean ones, this guidance is falsified for that model; on RLVR-trained models the clean variants win or tie at lower cost.
* **Production Reality & Tradeoffs:** Thinking inflates p99 latency and output-token spend; cap budgets per route and stream so users see progress. Several providers surface only summarized thinking, so never treat the visible trace as an audit log or parse it as the answer. Effort routing needs its own eval — misrouting hard tasks to minimal effort fails silently as fluent, wrong answers rather than as errors.
* **Red Flag:** Porting a pre-reasoning prompt library — ToT harnesses, "think step by step", few-shot CoT — verbatim onto a reasoning model, paying twice for deliberation, without ever A/B-testing it against a plain prompt plus the effort knob.
