### Q: How do you manage large tool catalogs in context: schema token cost, dynamic tool selection, and MCP tool bloat?
* **Difficulty:** Senior
* **Category:** Context Management
* **The 10-Second Pitch:** Every tool definition — name, description, JSON Schema — is prompt tokens on every request, and a handful of MCP servers can inject tens of thousands of schema tokens and dozens of near-duplicate tools before the first user turn, hurting both cost and selection accuracy. Fix it with a tiered catalog: a small cache-friendly static core, search or deferred loading for the long tail, and governance that eval-gates each new server as a context and accuracy tax.
* **The Deep Dive:** Token mechanics first: tool definitions are serialized into the prompt (effectively the system region), so the model pays prefill for the whole catalog whether or not any tool is called. A realistic definition — a description paragraph plus a JSON Schema with typed parameters — runs 150 to 800 tokens; MCP servers commonly expose 10 to 50 tools each, so five servers can exceed 50k schema tokens per request. The arithmetic:

  ```text
  catalog: 5 MCP servers x ~20 tools x ~500 tokens        = ~50,000 schema tokens
  static full catalog : 50k prefill every request (cacheable, but still read-cost
                        plus attention load on every turn)
  deferred loading    : ~8 core tools (~4k) + tool-search meta-tool
                        + ~3 discovered schemas (~1.5k) loaded on demand
                        -> ~5.5k typical, roughly 85-90% schema-token reduction
  ```

  Accuracy degrades before cost does. Tool selection is a classification problem over the catalog, and error grows with size and semantic overlap: three servers each exposing a "search" tool produce wrong-tool calls and mis-filled arguments, while irrelevant schemas act as distractors diluting attention over the instructions that matter. Function-calling benchmarks consistently show accuracy dropping as distractor tools are added, and Anthropic reported that replacing full MCP catalogs with on-demand tool search raised its MCP-heavy eval scores substantially (Claude Opus 4 from roughly 49 to 74 percent) while cutting schema tokens on the order of 85 percent. Levers in impact order: namespaced, non-overlapping names (`github_search_issues`, not `search`), descriptions stating when to use the tool and when not to, and consolidating chatty low-level tools into fewer high-level operations.

  Dynamic selection has two shapes. RAG-over-tools embeds tool descriptions offline and retrieves top-k schemas per request — cheap, but selection happens outside the model, so a bad retrieval silently removes a needed tool. Deferred loading keeps only a search meta-tool in context; the model queries the catalog mid-conversation and matched schemas are appended (the pattern behind Anthropic's Tool Search Tool). The caching interaction is the trap: tool schemas sit in the cached prefix, so a per-request dynamic set changes the prefix bytes and busts the cache — you can lose more to cache-write premiums than the pruned schemas saved. Resolutions: fix the set per session or route, not per message; order static core first, dynamic tail last, so changes invalidate only the tail; or prefer deferred loading, where discovered schemas arrive as later-context messages and the prefix stays byte-stable. Falsifiable test: A/B the full catalog against the dynamic scheme on a tool-use eval, tracking selection accuracy, cache-read tokens from the usage fields, and cost per solved task — if the dynamic scheme does not win on cost per solved task, the added machinery is falsified for that workload.
* **Production Reality & Tradeoffs:** MCP catalogs drift: servers add tools in upstream releases and your context bill grows without a deploy on your side, so pin server versions, log per-tool call frequency, and prune near-zero-usage tools. Set a token budget per server and lint descriptions in CI (length caps, required "use when" clause, duplicate-name detection). Retrieval-based selection adds an infra dependency and a new failure mode — needed-tool-not-retrieved — visible as hallucinated tool names or claimed inability; monitor unknown-tool-call rates. Deferred loading adds a discovery round-trip on cold paths.
* **Red Flag:** Treating tool definitions as free because "the model only reads them when calling tools" — they are prefill on every request — or connecting every available MCP server by default; never mentioning that dynamic tool sets invalidate the prompt-cache prefix signals this was never run in production.
