### Q: Compare provider-native structured outputs and tool-schema extraction with prompt-based JSON plus validation.
* **Difficulty:** Mid
* **Category:** Implementation
* **The 10-Second Pitch:** Strict schema modes use constrained decoding to guarantee syntactically valid, schema-conformant JSON — and guarantee nothing about whether the values are right. Tool-schema extraction gets the same structure through function-call plumbing; prompt-based JSON with a validator and a bounded repair loop is the portable fallback and often suffices for small, tolerant schemas.
* **The Deep Dive:** What strict modes actually guarantee: during decoding the sampler masks every token that would violate the compiled grammar, so the output always parses, contains every required field with the right types, and carries no trailing prose. What they do not guarantee: semantic correctness. The model can emit the wrong date, an entity from the wrong paragraph, or an empty string that legally satisfies a string type; an enum forces a choice even when no option applies, unless you add an explicit escape value like `"unknown"`. Schema-feature gaps compound this: strict implementations support a JSON Schema subset — numeric ranges, string patterns and formats, deep `anyOf` nesting, and recursion are commonly limited or ignored, and some dialects require all fields listed as required with `additionalProperties` set to false — so a downstream validator must still recheck the constraints the decoder never saw.

  Tool-call extraction predates native structured outputs: define the target shape as a function parameter schema, force that tool choice, and read the arguments. It behaves like strict mode where the provider constrains arguments, integrates naturally with agent loops, and works on providers without a dedicated response-format feature — with the same semantic gaps. Prompt-based JSON plus validation is the portable baseline: instruct the format with a short example, parse (stripping code fences), validate with Pydantic or Zod, and on failure re-prompt with the previous output plus the exact validator error, bounded at two or three attempts before dead-lettering. The error message is the active ingredient — most format failures fix in one fed-back retry. One more mechanic: forcing constrained JSON during reasoning-heavy extraction can measurably degrade answer quality by distorting the token distribution mid-thought, so either reason first and format in a second call, or order the schema with a free-text `reasoning` field before the answer fields.

  When plain prompting suffices: modern frontier models emit valid JSON for flat, sub-dozen-field schemas at well above 99 percent rates when prompted with an example, so if downstream tolerates a rare retry, prompting plus validation is simpler and provider-portable. Falsifiable test: run 500 extraction samples both ways. Strict mode will show a 100 percent parse rate by construction; if the claim is that it "fixed extraction quality," compare field-level accuracy against gold labels — parse rate and field accuracy move independently, and finding equal accuracy with a perfect parse rate falsifies quality claims while confirming the reliability one.
* **Production Reality & Tradeoffs:** Strict modes add first-request latency while the grammar compiles (then cache per schema), and their dialect differences create provider lock-in in your schema definitions. Repair loops add tail latency and cost, so alert on retry rate. Track syntactic validity and semantic accuracy as separate metrics; only the first is ever guaranteed.
* **Red Flag:** Treating "valid JSON" as "correct extraction" — enabling strict mode and deleting the semantic validators — or running a retry loop that never feeds the validator error back, which just resamples the same failure.
