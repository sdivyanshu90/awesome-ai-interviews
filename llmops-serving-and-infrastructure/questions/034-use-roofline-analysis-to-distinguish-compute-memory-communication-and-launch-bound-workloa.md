### Q: Use roofline analysis to distinguish compute, memory, communication, and launch-bound workloads.
* **Difficulty:** Principal
* **Category:** Performance
* **The 10-Second Pitch:** Compute arithmetic intensity—FLOPs per byte—and compare achieved performance with hardware compute and bandwidth roofs. Extend analysis with interconnect and launch/CPU limits to locate the active bottleneck.
* **The Deep Dive:** Arithmetic intensity is $I=useful FLOPs/bytes transferred at the memory level being analyzed$. For peak compute $C_peak$ and sustainable bandwidth $BW$, attainable performance is $P≤min(C_peak,BW·I)$; the ridge point is $I*=C_peak/BW$. If $I<I*$, reducing bytes or increasing reuse matters; if $I>I*$, tensor-core occupancy, instruction mix, and algorithmic FLOPs matter. Construct separate ceilings for HBM, L2/shared memory, PCIe/NVLink, and network. Distributed runtime is additionally bounded by collective volume/bandwidth plus latency and synchronization; a simple step model is $T≈max(T_compute,T_HBM,T_comm)$ only when overlap is real, otherwise exposed components add.

  Count bytes at the measured boundary after tiling/cache reuse, not the sum of tensor sizes in source code. Prefill’s large GEMMs reuse weights/activations and often move rightward toward the compute roof. Single-token decode repeatedly streams weights and a growing KV cache, so it often sits near the HBM roof. Tiny kernels can be far below both roofs because launch latency, dependency gaps, low occupancy, divergence, or CPU scheduling dominates; this is a latency/launch ceiling, not evidence against roofline.
* **Production Reality & Tradeoffs:** Vendor peaks assume particular precision, sparsity, clocks, and shapes; use sustained microbenchmark roofs on the deployed SKU/topology. Validate with achieved FLOPs, DRAM/L2 bytes, memory throughput, tensor-core utilization, occupancy, launch gaps, collective traces, and clock/power data. Kernel fusion raises effective intensity by eliminating intermediate traffic; quantization lowers bytes only if conversion and kernel support do not become the new limit. Optimizing FLOPs cannot fix bandwidth-bound decode.
* **Red Flag:** Calling low GPU utilization proof of compute bottleneck.
