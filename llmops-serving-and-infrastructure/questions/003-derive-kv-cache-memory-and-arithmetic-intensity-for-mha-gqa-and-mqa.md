### Q: Derive KV-cache memory and arithmetic intensity for MHA, GQA, and MQA.
* **Difficulty:** Senior
* **Category:** Math
* **The 10-Second Pitch:** KV bytes are $2·L·B·T·H_kv·d_h·bytes$, so MHA uses $H_kv=H_q$, GQA fewer grouped KV heads, and MQA one. Decode attention reads approximately this growing cache per step, making it bandwidth-heavy.
* **The Deep Dive:** For $L$ layers, batch/concurrent sequences $B$, retained length $T$, KV heads $H_kv$, head dimension $d_h$, and element width $s$ bytes, cache storage is $M_KV=2LBTH_kv d_h s$; factor two is key plus value. Per sequence per token it is $2LH_kv d_h s$. MHA has $H_kv=H_q$; GQA has $H_kv=H_q/g$ for query-group size $g$; MQA has one KV head. Example: $L=32$, $H_q=32$, $d_h=128$, BF16, and $T=8192$ consumes about $4 GiB$ per MHA sequence but $0.5 GiB$ with eight KV heads, before block waste and runtime buffers.

  For one decode token, QK dot products and weighted-value accumulation each cost approximately $2BH_qTd_h$ FLOPs under the common multiply-add-as-two convention, so attention is about $4BH_qTd_h$ FLOPs. KV reads are roughly $2BTH_kv d_hs$ bytes per layer if cache reuse is negligible. The cache-only arithmetic intensity is therefore approximately $2(H_q/H_kv)/s$ FLOP/byte: near $1$ for BF16 MHA, $g$ for $g$-way GQA, and $H_q$ for MQA. Projection weights and outputs add traffic, and kernels can reuse/broadcast one KV head across its query group, but the derivation explains why decode attention is bandwidth-sensitive and why GQA improves capacity and intensity.
* **Production Reality & Tradeoffs:** Real allocation rounds tokens to blocks, aligns tensors, reserves workspace, and competes with weights, activations, graph pools, and temporary logits. Prefix sharing and copy-on-write reduce effective $B·T$; beams and speculation increase it; sliding-window attention caps retained $T$. KV quantization reduces $s$ but adds scale metadata, conversion cost, and long-context quality risk. Admission must use p99 prompt plus output growth, not current length or parameter count alone.
* **Red Flag:** Using parameter count alone to estimate serving concurrency.
