### Q: Why do modern LLMs add multi-token-prediction heads during pretraining, and how does MTP interact with speculative decoding at serving?
* **Difficulty:** Senior
* **Category:** Training
* **The 10-Second Pitch:** MTP adds auxiliary heads that predict tokens $t+2, \dots, t+1+D$ at every position, densifying the training signal and forcing hidden states to pre-plan beyond the next token; at serving the extra head doubles as a nearly free self-speculation draft whose acceptance rate is high precisely because it shares the trunk's exact context.
* **The Deep Dive:** Standard next-token training gives one prediction per position; MTP attaches $D$ extra prediction depths so each position also predicts further-ahead tokens, multiplying supervised signals per forward pass and regularizing representations toward information that stays useful several tokens out. The total loss is the main cross-entropy plus a weighted average of the depth losses,

  $$
  \mathcal{L} \;=\; \mathcal{L}_{\mathrm{CE}}^{\mathrm{main}} \;+\; \frac{\lambda}{D} \sum_{k=1}^{D} \mathcal{L}_{\mathrm{CE}}^{(k)},
  $$

  where DeepSeek-V3 used $D = 1$ with $\lambda = 0.3$ early in training and $0.1$ later. The original parallel formulation (Gloeckle et al., 2024) hangs $D$ independent heads off the final trunk state, predicting token $t+k$ unconditionally on the intermediate ones. DeepSeek-V3's sequential MTP module is stronger: depth $k$ is a single extra transformer block that takes the RMSNorm'd representation from depth $k-1$ concatenated with the RMSNorm'd embedding of the token at offset $k$, passes them through a linear projection and the block, and reuses the shared embedding matrix and shared output head. Because each depth conditions on the true intervening token, it preserves the chain rule along the prediction path instead of predicting future tokens independently — which is exactly what makes it a well-calibrated draft later. Ablations showed consistent benchmark gains from the densified objective alone; the module can simply be discarded at inference.

  The serving interaction is self-speculative decoding. Keep the MTP block at inference: each decode step emits the main token plus a draft for the following position; the next forward pass verifies the draft (it computes the main head's distribution at that position anyway), accepting on match under greedy decoding or via the standard rejection-sampling rule under sampling — so the output distribution is provably unchanged. The economics are an acceptance-rate calculation. With one draft token and acceptance rate $a$, expected tokens per target forward is $1 + a$, and the extra cost is one transformer block on top of 61, so

  $$
  \text{speedup} \;\approx\; \frac{1 + a}{1 + \varepsilon}, \qquad a \approx 0.85\text{-}0.90,\ \varepsilon \approx 0.02\text{-}0.05 \;\Rightarrow\; \approx 1.8\times
  $$

  matching DeepSeek's reported 85-90 percent second-token acceptance and about 1.8x TPS. Compare a separate draft model: it needs its own weights, KV cache, and tokenizer alignment, and its acceptance suffers because it conditions on its own weaker representation of the context; the MTP draft reads the trunk's actual hidden state, which is why acceptance is so high. The falsifiable test is distribution preservation: emit MTP drafts directly without verification and you will measure divergence from the base model's outputs; with verification, per-token distributions must be statistically indistinguishable — if they are not, the acceptance rule is implemented wrong. The limiting case is $a \to 0$ (e.g., high-entropy creative sampling at high temperature): speedup collapses to $1/(1+\varepsilon)$, slightly worse than baseline, so speculative paths should disable themselves when measured acceptance drops.
* **Production Reality & Tradeoffs:** MTP costs extra pretraining memory and compute per step (one more block plus an extra output-head GEMM against a large vocabulary), and the loss weight needs annealing — too high and it taxes next-token quality. At serving, verification widens decode steps from one position to two, which is nearly free in the memory-bandwidth-bound regime but interacts with batching: mixed accept/reject across a batch fragments step lengths, and schedulers must handle variable tokens-per-step. Acceptance is domain-dependent — high on code and boilerplate, lower on high-temperature prose — so meter it per workload rather than trusting a headline 1.8x.
* **Red Flag:** Treating MTP heads as a way to emit several tokens per step without verification (that changes the output distribution), or describing parallel independent heads when the deployed design is sequential and conditions each depth on the true intervening token — the independence assumption is exactly what the sequential module fixes.
